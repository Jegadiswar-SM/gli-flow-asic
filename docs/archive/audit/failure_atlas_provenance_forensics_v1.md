## PHASE 14 — Signature Origin Trace

### Mechanism of Detection

The `signature_engine.py:scan_file()` function (`signature_engine.py:61-87`) uses a TWO-STEP detection:

1. **Exact regex match** (line 73-77): Escapes the `observed_signature` text and searches for it as a literal regex in the log. **Result: ZERO matches** — none of the 21 `observed_signature` texts exist verbatim in any log file.

2. **Category-keyword fallback** (line 79-82): If exact match fails, generates keywords from the signature's `category` field using `_get_keywords()` (line 90-104). These are broad, generic keywords:
   - `timing` → `["wns", "tns", "slack", "setup", "hold", "violat"]`
   - `drc` → `["drc", "violation", "spacing", "width", "enclosure", "antenna"]`
   - `congestion` → `["overflow", "congestion", "density"]`
   - `power` → `["ir drop", "voltage", "power"]`
   - `logic` → `["latch", "inferred", "combinatorial loop", "module.*not found"]`
   - `library` → `["not found", "missing", "can't open"]`
   - `floorplan` → `["aspect ratio", "core boundary", "pin.*place"]`
   - `routing` → `["open net", "short", "unconnected"]`

   **Result: 19/21 signatures match** via keyword fallback because common EDA terms like "tns", "violation", "power", "latch", "overflow" appear in any normal OpenROAD log.

   **The 2 non-matching signatures** (FA-0013, FA-0014) have FLOORPLAN category keywords that don't appear in the log.

### The `observed_signature` Text is Fabricated

Every log-signature entry's `signature` field stores the exact `observed_signature` text from `signatures.json`. This text is a **predefined example string, not actual log content**. For example:

| FA ID | DB `signature` value | Actually in log? |
|-------|---------------------|------------------|
| FA-0001 | `"WNS: -0.45  TNS: -12.3"` | NO (actual: WNS=0.0, TNS=0.0) |
| FA-0005 | `"Peak congestion 94% at (120, 340), overflow 320"` | NO (log mentions congestion but not these values) |
| FA-0019 | `"Density 97.3% in region (200, 300)-(400, 500), 32 cell overlaps"` | NO (log mentions density but not these values) |
| INF-PWR-001 | `"power_pass=True with all power metrics None/0.0 in telemetry"` | NO |

### Confirmatory Evidence

**Reproduction test** (Phase 16): Re-running `scan_file()` against the same artifacts reproduces all 19 log-signature entries. The `signature` field in reproduced entries would contain the same `observed_signature` text, confirming the DB entries are deterministic outputs of the keyword-fallback algorithm — not actual failure detections.

## PHASE 19 — Final Answers

### 1. Can current code reproduce the detections?
**YES, 25/26 entries are reproducible** — but the 19 log-signature entries are reproduced through a **keyword-fallback false positive mechanism**, not through actual failure detection. This is a correctness bug, not a reproducibility bug.

### 2. Did schema drift occur?
**NO significant drift.** The `observed_signature` key has been stable since commit `9cef34a1`. The `signatures/` subdirectory was added later but has no functional impact (its stubs have no `observed_signature` and are skipped). The detection algorithm changed from exact-only to exact+keyword-fallback in commit `c24a280`.

### 3. Did synthetic contamination occur?
**NO.** The `failure_injector.py` and `campaign_runner.py` tools exist but are separate from the production pipeline. They do not write to the `failure_atlas_entries` table.

### 4. Are the atlas records trustworthy?
**PARTIALLY.** 7 records (27%) are trustworthy — they have real evidence, specific measurements, and traceable code paths. 19 records (73%) are untrustworthy — they are keyword-fallback false positives with fabricated `observed_signature` text stored as the detection signature.

### 5. Can these records be used for GLI-SDI training?
**Only the 7 TRUSTED records should be used for training.** Including the 19 UNTRUSTED records would introduce severe false-positive contamination — the model would learn to associate normal EDA terminology (e.g., "tns" appearing in any log) with specific failure modes (e.g., timing violations), producing unreliable SDI assessments.

### Final Classification

# UNTRUSTWORTHY

> 73% of Failure Atlas records for this run are keyword-fallback false positives. The `observed_signature` field stores fabricated example text, not actual log evidence. The detection algorithm (exact regex → category-keyword fallback) generates detections based on broad EDA terminology present in any normal tool log, not on actual failure pattern matches.

### Required Actions

1. **Replace the keyword-fallback mechanism** in `signature_engine.py:79-82` with actual regex/pattern matching against real failure signatures. The current fallback creates an unacceptable false-positive rate.
2. **Validate each of the 19 UNTRUSTED entries** against actual log content — determine if the claimed failure genuinely exists or if the detection was purely keyword-based.
3. **Add a `detection_method` field** to the `failure_atlas_entries` table (or as a discriminator in `evidence`) to distinguish between EXACT_MATCH, KEYWORD_FALLBACK, METRIC_BASED, and INFRASTRUCTURE detections.
4. **Reject or retag** all 19 UNTRUSTED entries before using the Failure Atlas for GLI-SDI training data.
